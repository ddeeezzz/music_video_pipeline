# 模块A V2 非人声重拍切分打分说明

## 文档目的

这份文档说明 `module_a_v2` 在处理 `inst / chant / silence` 这三类非歌词窗口时，为什么会在某个 downbeat 上切开，以及相关分数是怎么来的。

对应实现文件：

- `src/music_video_pipeline/modules/module_a_v2/timeline/role_merger.py`

本文只描述代码中已经落地的事实，不额外补充未实现的策略。

## 1. 这套逻辑解决什么问题

当一个非歌词窗口过长时，模块 A 不会直接把整段原样保留，而是会尝试在窗口内部的 downbeat 上寻找更合理的切点。

当前版本的入口条件是：

- 窗口角色不是 `lyric`
- 窗口时长超过 `long_other_split_min_bars * bar_length_seconds`

对应代码位置：

- `_split_long_other_windows_by_major(...)`

## 2. 候选切点从哪里来

候选切点不是任意 beat，也不是任意 onset，而是窗口内部的 `major` beat。

在当前工程里，`major` 的来源是 allin1 的 `beat_position == 1`，也就是小节首拍 / downbeat 语义。代码在构建 `ModuleAOutput.beats` 时会把它映射成：

- `type = "major"`: 小节首拍
- `type = "minor"`: 其他拍位

对应代码位置：

- `src/music_video_pipeline/modules/module_a_v2/backends/allin1.py`
- `_build_module_a_beats_from_allin1(...)`

## 3. 为什么是“滑动桶”

代码不是在整个长窗口里一次性只选一个最强 downbeat，而是按一个滑动时间桶逐段向前推进。

当前关键参数：

- `long_other_split_min_bars`
  - 超过多少小节才启动细分
- `major_split_step_bars`
  - 每次往前看的桶宽，单位是“小节”

例如：

- 如果 `bar_length_seconds = 4.0`
- `major_split_step_bars = 2.5`

那么每个滑动桶宽度就是 `10` 秒。

流程是：

1. 从窗口起点开始，取一个 `2.5` 小节宽的时间桶。
2. 在这个桶里的所有 downbeat 候选中打分，挑一个。
3. 把这个 downbeat 当成新的锚点。
4. 再从新的锚点往后取下一个桶，重复。

这样做的作用是：

- 不会让超长非人声段只切一刀后剩下一大坨
- 又不会因为纯局部贪心而在很短时间里连续乱切

## 4. 打分用了哪些特征

每个候选 downbeat 会计算四类信号：

### 4.1 onset 能量峰值

看 beat 附近半窗内的 onset 峰值能量，字段名：

- `energy_raw`
- `energy_norm`

它回答的问题是：

- 这个 downbeat 附近有没有明显的节奏打点

### 4.2 onset 前后变化量

看 beat 前半窗和后半窗的 onset 峰值差异，字段名：

- `delta_raw`
- `delta_norm`

它回答的问题是：

- 这个 downbeat 前后，节奏冲击有没有发生变化

### 4.3 chroma 差异

只对伴奏 chroma 做前后半窗比较，计算前后中位 chroma 向量的余弦距离，字段名：

- `chroma_delta_raw`
- `chroma_delta_norm`

它回答的问题是：

- 这个 downbeat 前后，和声/音高集合有没有变化

这对 `inst` 特别重要，因为器乐段很多时候不是靠音量突变分段，而是靠和声或织体变化分段。

### 4.4 F0 差异

看 beat 前后半窗的中位音高差，字段名：

- `f0_delta_raw`
- `f0_delta_norm`

它回答的问题是：

- 这个 downbeat 前后，旋律音高是否发生明显跳变

对 `chant` 来说，这个信号权重更高。

## 5. `inst` 和 `chant` 为什么不是同一套权重

代码里有两套权重：

### `inst` 权重

- `chroma_delta`: `0.55`
- `onset_delta`: `0.25`
- `energy`: `0.20`
- `f0_delta`: `0.00`

解释：

- 器乐段更关注和声/配器变化，所以 `chroma` 占比最高
- `F0` 对伴奏切分帮助有限，因此直接不给权重

### `chant` 权重

- `chroma_delta`: `0.10`
- `onset_delta`: `0.25`
- `energy`: `0.15`
- `f0_delta`: `0.50`

解释：

- 吟唱段更像“有人声旋律但无明确歌词”，因此音高变化更重要
- 所以 `F0` 是主导项

## 6. `chant` 的 F0 为什么还分 `vocals / no_vocals / fallback`

`chant` 不直接固定用某一路 F0，而是先比较局部 RMS：

- 如果人声 RMS 明显高于伴奏 RMS
  - 优先用 `vocals` 的 F0
- 否则
  - 优先用 `no_vocals` 的 F0
- 如果首选路没有有效 F0
  - 再回退到另一条路

最后会把来源写进：

- `split_pitch_source`

这样做的原因是：

- 有些“吟唱”更接近主旋律线
- 有些段落虽然分类成 `chant`，但局部上真正稳定的音高信息反而在伴奏侧

## 7. 为什么还要乘一个距离惩罚

每个 beat 的综合分数不是直接把各特征相加，而是还要乘 `distance_penalty`。

这个惩罚来自：

- beat 到最近 onset 的距离

如果一个 downbeat 离最近 onset 太远，就算它的其他信号看起来不错，也会被压分，甚至直接失去候选资格。

这样做是为了避免：

- 选中“理论上是小节首拍，但听感上根本没有落点支撑”的切点

相关字段：

- `nearest_onset_distance`
- `distance_penalty`
- `near_onset_candidate`

## 8. 候选是怎么选出来的

在一个滑动桶里，选点规则是：

1. 先过滤掉不是有效候选的 beat
2. 如果所有有效候选分数都接近 0
   - 回退到桶尾 downbeat
   - `split_major_pick_reason = "fallback_index"`
3. 否则
   - 选综合分最高的 beat
   - 平分时优先离桶中心更近的 beat
   - 再平分时优先更早的 beat
   - `split_major_pick_reason = "energy_peak"`

这里的 `"energy_peak"` 名字有点历史包袱，实际并不只看 onset 能量，而是看整套综合分。

## 9. 最终会把哪些解释信息写回窗口

每个切开的子窗口都会带上一批解释字段，方便排查：

- `split_step_bars`
- `split_basis`
- `split_major_pick_reason`
- `split_pick_beat_type`
- `split_beat_energy_raw`
- `split_beat_delta_raw`
- `split_beat_chroma_delta_raw`
- `split_beat_f0_delta_raw`
- `split_pitch_source`
- `split_beat_score_base`
- `split_beat_score`
- `split_beat_score_components`
- `split_beat_onset_distance`
- `split_beat_distance_penalty`

所以如果后面你要检查某个非人声段为什么切在某个点，直接看这些字段就够了。

## 10. 这次改动后的当前配置语义

当前这轮实现里，推荐理解为：

- 歌词长句上限：由 `long_lyric_resplit_max_bars` 控制
- 非歌词长窗细分触发：由 `long_other_split_min_bars` 控制
- 非歌词 downbeat 滑动桶宽：由 `major_split_step_bars` 控制
- 小段视觉提前：由 `visual_lead_seconds` 控制

这几项都已经从硬编码抽成了正式配置字段。
